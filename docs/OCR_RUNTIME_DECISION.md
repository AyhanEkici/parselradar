# OCR RUNTIME DECISION

## Phase
P2.OCR-DEPENDENCY-1 - OCR Runtime Selection + Local Proof-of-Concept

## Selected Decision
`DEFER_RUNTIME_OCR`

## Decision Summary
Runtime OCR is deferred in this phase. The platform keeps filename/manual suggestion behavior and does not activate OCR extraction runtime yet.

## Why Runtime OCR Is Deferred
- No OCR runtime dependency is currently present in `apps/web` or `apps/api`.
- Browser OCR with `tesseract.js` is privacy-friendly but adds bundle/runtime cost and language-data handling that should be validated with explicit performance budgets.
- Server-local OCR increases deployment/runtime footprint and operational complexity before a dedicated packaging/runtime plan exists.
- Current product constraints require strict safety boundaries, no external sharing, and no official-proof claims.

## Option Analysis
1. Browser/local OCR (`tesseract.js`):
- Pros: local processing, no external API requirement, aligns with privacy goals.
- Cons: bundle size, client CPU usage, worker/language pack loading strategy, UX latency on low-end devices.

2. Server-side local OCR:
- Pros: centralized execution and easier policy control.
- Cons: heavier deployment dependencies, increased infra footprint, maintenance overhead.

3. External OCR/AI API:
- Rejected for this phase.
- Requires separate privacy/security/legal approval and explicit data-handling policy.

4. Defer runtime OCR:
- Chosen now to preserve controlled scope and maintain build/runtime stability.

## Privacy and Security Constraints
Any future OCR runtime must satisfy all of the following:
- No external OCR/AI API by default.
- No third-party upload of user images/documents.
- No automatic official verification.
- No legal/tapu/imar proof claims generated by OCR output.
- OCR output stays suggestion metadata only.
- Manual user/admin confirmation remains required.
- No user-facing buy advice.
- No admin BUY/WATCH runtime activation from OCR text in this phase.

## Recommended Future Dependency Path
Primary candidate for local-only browser POC: `tesseract.js` with lazy loading.

Implementation notes for a future scoped POC:
- Dynamic import only when user explicitly clicks "Try OCR preview".
- Restrict to image files for first iteration.
- Show extracted text as `OCR preview` only.
- Show confidence when available.
- Always display `Manual confirmation required`.
- Keep evidence/source fields editable and user/admin-overridable.
- Do not auto-persist OCR text to backend unless separately approved.

## Acceptance Criteria Before Real OCR Activation
All criteria must pass before enabling OCR runtime:
- Dependency decision approved and documented.
- Web/API builds pass with OCR code path enabled.
- Performance budget defined and validated for OCR execution.
- OCR UI clearly labeled as suggestion-only with manual confirmation gate.
- No external OCR/API network calls in runtime path.
- No change to official verification posture.
- No user-facing investment advice behavior introduced.
- Security/privacy review completed for OCR processing path.

## Non-Goals (This Phase)
- No real OCR extraction runtime.
- No backend OCR processing contract activation.
- No automatic evidence/source assignment from OCR text.
- No buy/watch opportunity runtime changes.

## Status
Public launch readiness remains `NOT_READY`.
